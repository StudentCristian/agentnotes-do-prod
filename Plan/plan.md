## Plan: AgentNotes DO Rollout

Diseñar AgentNotes como una sola aplicación web de Next.js en DigitalOcean App Platform, con desarrollo local en devcontainer y despliegues separados de staging y production usando el mismo app spec parametrizado. La base de datos irá en Managed PostgreSQL con bindable variables; el audio temporal usará Spaces con presigned URLs, eliminación explícita tras transcripción y lifecycle rule de respaldo; el despliegue se controlará por GitHub Actions para no exponer secretos ni depender del panel.

**Steps**
1. Fijar la base de arquitectura y los límites del rollout a partir del repo actual. Reusar `package.json`, `next.config.ts`, `app/api/transcribe/route.ts`, `lib/bamlClient.ts`, `baml_src/transcription.baml`, `components/audio/audio-bottom-bar.tsx` y `.github/skills/copilot-instructions.md` como referencias obligatorias. Esto define que la app es un único servicio Next.js con API routes embebidas, no frontend y backend separados.
2. Etapa 0, devcontainer con paridad de producción. Crear `.devcontainer/devcontainer.json` y sus archivos auxiliares para Node 20, pnpm, perfil local con PostgreSQL y almacenamiento S3-compatible local, carga de variables desde `.env.local`, terminal y volúmenes compatibles con WSL2, y post-create setup para instalar dependencias y validar `pnpm build`. Este paso prepara development y bloquea las etapas de validación local. *Bloquea 6 y 7; puede diseñarse en paralelo con 3, 4 y 5.*
3. Diseñar `.do/app.yaml` para App Platform como un único componente `service` tipo web. Definir `build_command` y `run_command` de Next.js, `http_port`, health check, `deploy_on_push: false`, variables `DATABASE_URL`, `DATABASE_HOST`, `DATABASE_PORT`, `GOOGLE_AI_API_KEY`, `DO_SPACES_KEY`, `DO_SPACES_SECRET`, `DO_SPACES_BUCKET`, `DO_SPACES_REGION`, y dominios/ingress para staging y production. Modelar development como entorno local fuera de App Platform; staging y production se despliegan como apps separadas o como el mismo spec apuntado por distintos `DO_APP_ID`. *Depende de 1; informa 5 y 7.*
4. Diseñar `db-setup.sql` para Managed PostgreSQL. Crear tablas `patients`, `consultations`, `consultation_fields`, `field_templates`, `generated_documents`, y agregar la tabla de soporte para plantillas de documentos si se quiere persistir el catálogo real de DOCX. Incluir extensiones mínimas, claves primarias UUID, índices básicos, claves foráneas, timestamps, y grants mínimos para el rol de aplicación. Mapear bindable vars a `DATABASE_URL`, `DATABASE_HOST=${db.HOSTNAME}`, `DATABASE_PORT=${db.PORT}` en el spec. Recomendar un clúster administrado por entorno real, con development local en contenedor. *Depende de 1; puede avanzar en paralelo con 5.*
5. Diseñar `spaces-cors.json` y la configuración de Spaces. Usar bucket `agentnotes-audio-tmp`, prefijo `audio/tmp/`, CORS con `PUT` y `OPTIONS` solo para `https://app.${APP_DOMAIN}` y `https://staging.${APP_DOMAIN}`, lifecycle rule de expiración a 1 día solo para ese prefijo, y acceso exclusivamente por presigned URLs sin ACL pública. Reflejar en `.do/app.yaml` únicamente variables de entorno; Spaces no se modela como componente bindable dentro del spec. *Depende de 3 para los orígenes finales y puede avanzar en paralelo con 4.*
6. Diseñar networking y dominios. Configurar el dominio en DigitalOcean Domains con HTTPS automático de App Platform, dominio primario `app.${APP_DOMAIN}` para production y `staging.${APP_DOMAIN}` para staging, y reglas CORS en el ingress de App Platform para `/api/*` incluyendo `OPTIONS`. Añadir al plan de implementación una allowlist de orígenes en las API routes sensibles para que la protección no dependa solo del ingress. *Depende de 3 y ajusta 5.*
7. Diseñar el alineamiento de código requerido por la infraestructura antes del despliegue útil. Sustituir el flujo actual `audio_text` por el flujo `objectKey + fieldsSchema`, crear `/api/audio/upload-url`, refactorizar `/api/transcribe` para generar presigned GET URL, invocar BAML con `Audio.from_url(url)`, eliminar el objeto con `DeleteObjectCommand` en `finally`, y reemplazar en `baml_src/transcription.baml` la función `TranscribeAudio` por `TranscribeConsultation(audio, fields_schema) -> ConsultationOutput`. Mantener `MediaRecorder.pause()` y `resume()` intactos en `audio-bottom-bar.tsx`, y sacar definitivamente cualquier dependencia de Web Speech API. *Depende de 2, 3, 4 y 6; bloquea 8.*
8. Diseñar CI/CD con `.github/workflows/deploy.yml`. Configurar un único workflow que despliegue a staging con pushes a `staging` y a production con pushes a `main`, usando `digitalocean/app_action/deploy@v2`, sin imprimir secretos, con `DIGITALOCEAN_ACCESS_TOKEN` y `DO_APP_ID` tomados desde GitHub Secrets por entorno, y con inyección de secretos adicionales para la sustitución del spec cuando corresponda. Añadir validaciones previas de `pnpm install`, `pnpm build`, `doctl apps spec validate` y, si el equipo lo acepta, `doctl app dev build` como gate local o en CI. *Depende de 3; conviene terminar 7 antes de habilitar despliegues automáticos.*
9. Generar el directorio `Plan/` con etapas staged y criterios de salida claros: `01-infra-base.md`, `02-audio-flow.md`, `03-baml.md`, `04-cicd.md`, `05-production.md`. Hacer que cada etapa enumere tareas, prerequisitos, comandos de verificación y artefactos esperados. Mapearlas a la secuencia pedida: Etapa 0 local/devcontainer; Etapa 1 infra base; Etapa 2 flujo de audio; Etapa 3 BAML; Etapa 4 CI/CD y staging; Etapa 5 validación y production. *Depende de 2 a 8 porque consolida la ejecución.*

**Decisions**
- Development no se modela como app de App Platform; se resuelve con devcontainer local. Los entornos cloud reales son staging y production con recursos separados y secretos separados.
- El app spec recomendado es uno solo, parametrizado y controlado por GitHub Actions; no se recomienda mantener specs divergentes por entorno salvo que aparezcan diferencias de red o sizing.
- PostgreSQL sigue el camino recomendado de bindable variables, no credenciales embebidas ni schema isolation multi-tenant para v1.
- Spaces se integra por variables secretas y presigned URLs; App Platform no provee bindable vars nativas para buckets.
- El bucket pedido puede ser único para staging y production solo si se segregan claves por prefijo y CORS por origen exacto; para menor riesgo operativo, sigue siendo preferible un bucket por entorno aunque no es requisito de esta entrega.
- Queda fuera de esta tanda la provisión real de credenciales, la ejecución contra recursos de DigitalOcean y la migración completa de plantillas DOCX desde EdgeStore si el usuario no la pide explícitamente.

**Further Considerations**
1. Añadir un endpoint de health check dedicado es recomendable aunque no forme parte de los artefactos pedidos; evita depender del comportamiento por defecto de Next.js para los reinicios de App Platform.
2. El binario `@boundaryml/baml-linux-x64-gnu` hace obligatorio validar el build real en Linux antes de staging; ese es el mayor riesgo técnico inmediato del despliegue.
3. Si más adelante se quiere persistir los DOCX generados, conviene ampliar `generated_documents` con `storage_provider`, `object_key` y `download_url` en lugar de guardar blobs en PostgreSQL.
